# 数据规模匹配论文 - 完整方案

## 🎯 目标

让我们的数据处理生成的数据量级与论文保持一致，避免数据规模过大带来的问题。

## 📊 问题分析

### 论文 vs 原始数据集的规模对比

| 数据集 | 原始训练样本 | 论文5-core后交互 | 差距 |
|--------|--------------|------------------|------|
| Video_Games | 736,800 | 153,221 | **4.8x** |
| Movies_and_TV | 7,100,000 | 136,471 | **52x** |
| Electronics | 1,400,000 | 197,984 | **7.1x** |
| Home_and_Kitchen | 2,500,000 | 256,001 | **9.8x** |
| Tools_and_Home_Improvement | 6,000,000 | 159,969 | **37.5x** |
| Arts_Crafts_and_Sewing | ? | 132,566 | - |

**关键发现**：
- 📈 Amazon原始数据集规模**远大于**论文使用的规模
- 🔍 论文数据量只是原始数据的**2%-30%**
- ⚠️ 如果不采样，生成的数据量会**超出论文数倍**

### 直接处理的问题

❌ **不采样直接处理**:
1. 处理时间：数小时甚至更长
2. 内存占用：GB级别，可能OOM
3. 生成数据量：Video_Games可能生成400K+样本（论文仅153K总交互）
4. 训练时间：数据量过大影响训练效率

## ✅ 解决方案：两阶段采样

### 方案架构

```
Step 0: 原始数据采样 (新增)
  ├─ 输入: intermediate/{Category}_interactions.parquet (数百万交互)
  ├─ 采样: 基于用户随机采样，保留30%左右用户
  └─ 输出: intermediate_sampled/{Category}_interactions.parquet
  
Step 1: 5-core过滤 (Notebook 02)
  ├─ 输入: intermediate_sampled/ (已采样)
  ├─ 过滤: 迭代式5-core过滤 + max_len=10
  └─ 输出: {Category}/5-core/downstream/train_data.txt
  
Step 2-4: 训练数据生成 (Scripts)
  ├─ 从train_data.txt完整扩展生成CSV
  └─ 生成IEM数据和验证
```

### Step 0: 采样脚本详解

**文件**: `scripts/step0_sample_raw_data.py`

**采样率设计** (针对各领域特点):

```python
SAMPLING_RATES = {
    'Video_Games': 0.30,              # 736K → 220K → 5-core后~153K ✓
    'Arts_Crafts_and_Sewing': 0.40,   # 需要保留更多用户
    'Movies_and_TV': 0.05,            # 7.1M → 355K → 5-core后~140K ✓
    'Home_and_Kitchen': 0.30,         # 2.5M → 750K → 5-core后~260K ✓
    'Electronics': 0.20,              # 1.4M → 280K → 5-core后~200K ✓
    'Tools_and_Home_Improvement': 0.15, # 6.0M → 900K → 5-core后~160K ✓
}
```

**采样原理**:
1. **基于用户采样** (而非基于交互)
   - 保留X%的用户
   - 保留这些用户的**所有**交互
   - 保持用户行为序列的完整性

2. **为什么这样设计？**
   - 5-core过滤会再次减少数据量（约30-50%）
   - 采样后仍能保证足够的交互密度通过5-core
   - 最终数据量接近论文规模

**预期效果**:

| 数据集 | 原始 | 采样后 | 5-core后 | 论文 | 匹配度 |
|--------|------|--------|----------|------|--------|
| Video_Games | 736K | 220K | ~153K | 153K | ✓ |
| Movies_and_TV | 7.1M | 355K | ~140K | 136K | ✓ |
| Electronics | 1.4M | 280K | ~200K | 198K | ✓ |

## 💡 技术要点

### 为什么不在5-core后采样？

**方案A** (❌ 不推荐): 5-core过滤后再采样
```
原始数据 → 5-core过滤 (耗时!) → 采样
```
问题: 
- 5-core过滤处理全量数据，耗时长
- 内存占用大

**方案B** (✅ 推荐): 5-core过滤前采样
```
原始数据 → 采样 → 5-core过滤 (快速!)
```
优势:
- 5-core只需处理30%数据，速度快5x
- 内存占用减少70%
- 最终结果等价（都通过5-core）

### 为什么基于用户采样而非交互？

**基于用户**:
```python
# 保留30%用户的所有交互
sampled_users = random.sample(all_users, n_users * 0.3)
df = df[df['user_id'].isin(sampled_users)]
```
✅ 保持序列完整性
✅ 符合推荐系统特性
✅ 5-core过滤效果好

**基于交互** (❌):
```python
# 随机保留30%交互
df = df.sample(frac=0.3)
```
❌ 破坏序列结构
❌ 5-core过滤后可能无数据
❌ 不符合实际场景

- ✅ 结果可复现

- ✅ 便于调试

### 3. 不同需求的采样率调整

**如果想要更接近论文规模**:
```python

```

```python
# 提高采样率
SAMPLING_RATES['Video_Games'] = 0.40  # 从0.30提高到0.40
```
